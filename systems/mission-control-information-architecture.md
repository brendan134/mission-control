# Mission Control Information Architecture

## Purpose
Define the structure of Mission Control so the AI operating system is easy to use, maintain, and migrate.

---

## Design Principles
- practical first
- clear navigation
- reusable system assets
- minimal friction
- portable structure
- operator-first documentation

---

## Top-Level Sections

### 1. Overview
Purpose: quick understanding of the system and current state.

Include:
- system purpose
- current priorities
- active projects
- recent decisions
- key alerts or blockers

### 2. Operator
Purpose: help the operator run the system correctly.

Include:
- operator cheat sheet
- operator rules
- operating constraints
- escalation logic

### 3. Models
Purpose: make model selection and cost discipline simple.

Include:
- model policy
- model routing
- model aliases
- escalation rules
- cost-saving principles

### 4. Planning
Purpose: support weekly and quarterly execution.

Include:
- weekly planning template
- quarterly priorities
- current priorities
- planning cadence

### 5. Workflows
Purpose: give repeatable execution processes.

Include:
- task capture and prioritisation
- content production
- client workflow docs
- sales/admin workflows

### 6. Templates
Purpose: store reusable copy-and-use documents.

Include:
- planning templates
- note templates
- message templates
- SOP templates
- content templates

### 7. Projects
Purpose: track live work and major initiatives.

Include:
- current priorities
- project files
- status notes
- next actions
- decision records

### 8. Business
Purpose: store business context and positioning.

Include:
- Leader By Design overview
- offers
- audience notes
- business model notes
- positioning

### 9. System
Purpose: manage the AI operating system itself.

Include:
- docs index
- architecture notes
- system rules
- migration notes
- future improvements

---

## Recommended Navigation Order
1. Overview
2. Current Priorities
3. Operator Cheat Sheet
4. Weekly Planning
5. Workflows
6. Templates
7. Projects
8. Models
9. Business
10. System

---

## Rules for Docs in Mission Control
- each doc should have one clear purpose
- prefer short pages over long mixed pages
- write for action, not theory
- keep naming simple and consistent
- update files when system rules change

---

## Migration Rule
Mission Control should be a view over portable files, not the only place information lives.
The files remain the source of truth.
