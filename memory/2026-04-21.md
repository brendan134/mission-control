## Cloudflare Tunnel Troubleshooting (April 21, Evening)

### Problem
Tunnel kept dropping after container restarts. Every time OpenClaw container restarts on Hostinger VPS, tunnel goes down and needs recreation.

### Symptoms
- `ERR failed to serve tunnel connection error="control stream encountered a failure"`
- `ERR Connection terminated`
- Tunnel trying to reach `[::1]:3003` (IPv6) instead of `127.0.0.1:3003` (IPv4)
- External URL `mc.brendanrogers.au` returning 530

### Root Cause
- Cloudflare tunnel ingress rule was set to `http://localhost:3003`
- Inside container, `localhost` resolves to IPv6 `[::1]`
- Mission Control not accessible on IPv6, only IPv4

### Solution Applied
1. **Updated ingress rule** in Cloudflare Dashboard: `http://127.0.0.1:3003` (not localhost)
2. Updated token in `ecosystem.config.js`
3. Created startup script: `/data/.openclaw/start-tunnel.sh`
4. Used `nohup` to keep cloudflared stable

### Final Working State
- Mission Control: ✅ Running locally (127.0.0.1:3003)
- Cloudflare Tunnel: ✅ Connected and stable
- External URL: ✅ `mc.brendanrogers.au` returns 200

### Startup Script Created
Location: `/data/.openclaw/start-tunnel.sh`
- Kills existing processes
- Starts Mission Control with PORT=3003
- Starts cloudflared with nohup
- Verifies external connectivity

### Key Learning
Always use `127.0.0.1` not `localhost` in Cloudflare tunnel ingress rules when the service only binds to IPv4.
